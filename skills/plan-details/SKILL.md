## Step 2：逐论点深化（核心主循环）

> **对骨架中每一个论点，按以下 4 个子步骤依次进行。**
> 每次处理一个论点，处理完后问"继续下一个吗？"再推进。

---

### 2-A：证据与数据

**提问：**
> 这个论点「{论点}」，你有什么证据或数据来支撑它？
> （可以是监控截图数字、调研结论、线上事故复盘、实验结果等）

**分支处理：**

**如果用户说"有"：**
- 追问具体数字/来源
- 进行**空姐效应检查**：
  > 这个数据/结论，你的听众已经知道吗？
  > - 若已知 → 引导换角度或挖更深层的 Why（见深化提示词）
  > - 若未知 → 记录，标注为"有效证据"

**如果用户说"没有"或"不确定"：**
- 提供以下思考框架，让用户选一个继续：

**Qd-2：想用哪个框架来推导这个论点的支撑？**

```
A. 五个为什么 — 一层一层往下问"为什么会这样"，找到根因
B. MECE 拆解 — 把论点拆成不重叠、完全覆盖的子维度              ← 理性分析首选
C. 象限图 / 二元对立 — 找两个关键维度，四象限定位你的解法
D. AI 辅助深化 — 告诉 AI 你的初步想法，AI 帮你挖三层根因
```

> 💡 不确定时选 B（MECE）最安全；有明确现象但不知道原因时选 A（五个为什么）；想描述"我们的方案和别人不一样"时选 C。

  ```
  框架 A：五个为什么
  → 这个现象的原因是什么？这个原因的原因是什么？继续追问3-5层

  框架 B：MECE 拆解
  → 把这个论点拆成互不重叠、完全覆盖的子维度
  → 例：成本问题 = 人力成本 + 机器成本 + 时间成本

  框架 C：象限图/二元对立
  → 找出两个关键维度，画出四象限，你的解法落在哪个象限？

  框架 D：AI 辅助深化（见下方模板）
  ```

**AI 辅助深化提示词模板（内置，用于无证据时推导）：**

```markdown
# 问题：{论点}

## 表层原因：
{用户的初步回答}

## 更深层原因：
（为什么会有上面这个原因？）

## 更更深层原因：
（继续追问，直到触达系统性根因）

## 方法论：
（针对根因，有什么可复用的解决思路？）

## 进一步思考：
（这个方法在什么场景下不适用？有没有反例？）
```

### 2-B：真实案例

**提问：**
> 针对「{论点}」，你有没有工作中的**真实案例**？
> 不是行业通用案例，而是你自己亲身经历的、有细节的、有情绪的故事。

**如果有：**
AI 帮结构化为以下格式：

```markdown
**背景（Context）：**
[项目背景、当时的处境、规模数字]

**冲突（Conflict）：**
[出现了什么问题？为什么难？谁受到影响？]

**解决（Resolution）：**
[你怎么分析？做了什么决定？为什么选这个方案而不是其他？]

**结果（Result）：**
[量化数字 + 情景细节，听众能感受到"哦，真的解决了"]

**参考图片：**
[若 references/ 中有相关图片，AI 主动提示：「references/ 目录中有 [文件名]，这张图是否用于支撑这个案例？」]
- 若是 → 在这里记录：`![描述](references/文件名)`
- 若否 → 记录 `（暂无配图，如有相关截图可放入 references/ 目录）`
```

然后做**空姐效应检查**：
> 你想在 PPT 上放这个案例的哪个部分？
> 放在 PPT 上的内容，演讲时就不要再读一遍了。

**如果没有：**
> 行业通用案例也可以先用，但有一个风险：听众听过太多，效果打折。
> 你能不能想想，有没有哪个**踩过坑、走过弯路**的经历，哪怕是失败的？
> 失败案例往往比成功案例更有共鸣。

### 2-C：深度展开方式选择

**Qd-3：这个论点怎么展开？**

```
A. 局部金字塔 — 先结论，再给 3 个支撑理由，每个理由有 1 个事实
   适合：说明型论点（"为什么要这么做"）
B. 局部一波三折 — 挑战 → 思考 → 成功 → 更大挑战（至少 2 轮）  ← 分享型推荐
   适合：有情绪弧度的故事段落
C. F.A.B 展开 — Feature（做了什么）→ Advantage（技术优势）→ Benefit（业务价值）
   适合：介绍技术方案，避免只说 Feature 不说 Benefit             ← 汇报型推荐
D. 自定义：我有另一种展开方式（描述给 AI）
```

> 💡 不确定时：分享型选 B，汇报型选 C，纯逻辑说明选 A。也可以组合（比如先 A 再 B）。

```
选项 A：局部金字塔
→ 先说结论，再给3个支撑理由，每个理由有1个事实
→ 适合：说明型论点（"为什么要这么做"）

选项 B：局部一波三折
→ 挑战 → 思考 → 成功 → 更大挑战（至少2轮）
→ 适合：分享型故事段落，有情绪弧度

选项 C：F.A.B 展开
→ Feature（你做了什么）→ Advantage（技术/工程优势）→ Benefit（对听众/业务的价值）
→ 适合：介绍技术方案，避免只说 Feature 不说 Benefit
```

根据用户选择，AI 协助补充细节。

### 2-D：深度自检（AI 主动挑战）

> 在记录内容前，AI 做以下检查。有问题的项**必须追问**，没问题的项快速跳过。

**通用检查（所有类型）：**

| 检查项 | 问题 | 风险 |
|--------|------|------|
| What/How vs Why | 你讲的是做了什么，还是也解释了**为什么这样做**？ | 缺 Why = 没有深度 |
| 授人以渔 | 听众能把这个方法复用到自己的工作中吗？ | 无法复用 = 没有启发 |
| 距离感 | 这个内容是否太高屋建瓴，离听众实际工作太远？ | 太空 = 听众没感觉 |
| 假问题识别 | 描述问题时，有没有"不/没有/非"这样的字眼？ | 假问题 = 说的是解决方案，不是真问题 |
| 推导过程 | 你说的是"做了什么"，还是"为什么这样做、排除了哪些方案"？ | 缺推导 = 像是碰运气 |

**假问题识别举例（内置）：**
```
❌ 假问题：K8S 的资源无法复用，导致无法支持更多业务
✅ 真问题：成本压力下，资源有限但业务增长快
---
❌ 假问题：缺乏统一的监控平台
✅ 真问题：故障发生时，平均定位时长超过30分钟，影响业务损失每小时XX万
```

## Step 3：汇报型专项检查

> **仅当 ppt-hours.json 中 `type === "汇报型"` 时触发此步骤。**
> 每个论点完成 Step 2 后，追加以下检查：

### 3-A：成果 vs 事情

> 你描述的是**成果**还是**事情**？

```
❌ 事情：建设质量体系
❌ 事情：上线了监控平台

✅ 成果：通过构建质量保障体系，XX 应用持续保持零故障单（12个月）
✅ 成果：监控平台上线后，故障定位时长从30分钟降至3分钟，Q3节约人力成本XX万
```

### 3-B：痛点数字化

> 在说明问题时，能否把痛点**从定性变成定量**？

```
从不痛到很痛的陈述梯度：
  不痛：我们是个 web 服务，需要提升 SLO
  一般：我们是个核心 web 服务，涉及业务关键流程
  较痛：web 服务涉及客户支付流程
  很痛：三个9的服务质量 = 每月损失超过1000万
```

追问：你能把「{论点描述的痛点}」用一个数字来表达吗？

### 3-B+：成果真实性——引导截图进 references/

> 方法论提醒：夸张的数字（如可用性 100%、零缺陷）评委会怀疑真实性。
> 最有力的证明：监控截图 + 人证（客户/业务方认可截图）

**AI 提问：**
> 你描述的这个成果「{成果描述}」，有没有可以截图的证明？

**Qd-4：成果证明材料，你有哪些？**

```
A. 有监控看板截图 / 数据大盘截图              ← 最有说服力
B. 有业务方或客户的认可（截图/文字/邮件）
C. A + B 都有（双重证明）                    ← 推荐（最可信）
D. 暂时没有，但可以去找
E. 这个成果不需要额外证明（说明理由）
```

> 💡 评委看到监控截图的那一刻，质疑会直接消失。10分钟找一张截图，比口头解释30分钟有效。

**若选 A / B / C：**
> 「请把截图保存到 `~/.pptdog/projects/<slug>/references/` 目录下。
> 建议命名：`monitor-[描述]-[论点编号].png` 或 `proof-[描述]-[论点编号].png`
> 保存后告诉我文件名，我会记录到这个论点的 details.md 里。」

收到文件名后，在该论点的「参考图片」字段里记录：
```markdown
**参考图片：**
- `![监控截图](references/monitor-xxx.png)` — 证明成果真实性（SLO = 99.99%，截至XXXX年）
```

**若选 D：**
> 「建议在演讲前找到这张截图——没有图的数字在汇报中说服力打六折。
> 我在 details.md 里记录一个待办：`⚠️ 待补充：成果证明截图`，你准备好后回来更新。」

**若选 E：**
> 「好，说明一下为什么不需要额外证明？（AI 记录到 details.md 备注中）」

### 3-C："自己"的举证

> 这个成果，说的是**你**的贡献，还是团队整体的成果？

- 用"我"而不是"我们"
- 如果是多人合作，请说清楚：你负责的边界是什么？你的核心决策是什么？
- 在架构图上，明确**圈出**你负责的模块

### 3-D：技术难度描述

> 你在描述技术难点时，有没有体现**递进式深度**？

```
通常大家做到 A、B、C，
我们还做了 D（描述为什么 D 更难），
甚至做了 E（描述 E 的突破点）。
```

追问：行业里通常怎么做？你做的有什么不同？有没有参考过 Google/CNCF/QCon 等的做法？

### 3-E：运气牛检查

> 这个成果，有没有因果推导？还是像"碰运气"？

```
❌ 运气牛：我们在疫情期间做了这个，流量暴增，成功了
✅ 因果牛：疫情初期我判断流量会暴增，提前3周做了容量规划，
           具体依据是XX（数据/经验），最终顶住了X倍流量
```

追问：如果去掉外部因素（如政策/市场风口），你的成果还成立吗？为什么？

## 附录：方法论速查

### 空姐效应
PPT 上写了的内容，演讲时就不要读了。听众眼睛盯着屏幕就离不开了。
只在 PPT 上放：关键词、数字、图示说明——不是你要说的完整句子。

### 一波三折模板
```
挑战和失败：[产生共鸣的情景，听众在场的模样，自己的感受]
思考：[至少3层根因：原因 / 更深层原因 / 更更深层原因 + 推导逻辑 + 解决方案]
成功的结果：[量化数字 + 共鸣细节]
新挑战（递进，更难）：[同样有情景和细节]
再次思考：[与上次有递进关系，更深层更细节更本质]
成功的结果：[...]
最高潮挑战 → 最终成功：[...]
```

### F.A.B 法则
- **Feature（特性）**：你做了什么，技术实现是什么
- **Advantage（优势）**：相比现有方案，技术/工程上的优势在哪里
- **Benefit（价值）**：对业务、对用户、对团队的实际价值是什么

### 已知地图
听众通常知道：What（是什么）、How（怎么做）
听众通常不知道：**Why（更深入的为什么）**
→ 你的内容应该主要覆盖"Why"区域
