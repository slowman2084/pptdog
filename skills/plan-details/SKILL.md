# Skill: /plan-details — 素材整理

> **使用方式：** 运行 `/plan-details`。
>
> **位置：** 在 `/ppt-hours` 之后、`/plan-mindmap` 之前。
> 这一步不需要思维导图，思维导图是基于你整理好的素材来生成的，顺序不能反。
>
> **核心原则：先倒出，再分类，再找关系。**
> 不要在这一步想"这条素材放哪章"——那是 /plan-mindmap 的事。
> 这一步的问题只有一个：**你有什么？**

## Step 0：摸清底牌

AI 说：

> 「我们先不想结构，先把你有的东西全部列出来。
> 你现在准备这场演讲，有没有现成的材料？」

**Qd-0：你有什么现成材料？**

```
A. 有 PPT 草稿 / 文档 / 脑图——粘贴或告诉我在哪里      ← 有的话先看
B. 有一些零散想法，可以说给你听
C. references/ 里已经有文件了，从那里开始
D. 什么都没有，从主题开始想                            ← 选这个也没关系
E. A + B 都有（有材料，还有补充想法）
```

> 💡 不管选哪个，目标都一样——把你脑子里想讲的东西梳理清楚。别担心顺序和逻辑，那是后面的事。

**若选 A 或 E：**
让用户粘贴或描述现有材料。AI 快速浏览后提炼出素材清单的初稿，不做评价，继续进入 Step 1。

**若选 C：**
AI 读取 references/ 里的文件（图片描述、文档摘要），列出初步素材清单，继续进入 Step 1。

**若选 B、D：**
直接进入 Step 1。

---

## Step 1：倒出素材

AI 说：

> 「好，现在把你想在这场演讲里讲的东西全部倒出来。
> 不管顺序，不管完不完整，不管是不是已经整理好——想到什么说什么。
> 可以是：一个故事、一个数字、一个结论、一个踩过的坑、一件让你印象深刻的事……
> 说完了告诉我，我来帮你整理。」

用户自由输入后，AI 做以下工作（**不要打断用户，等用户说完了再整理**）：

1. 把用户说的内容整理成**编号素材清单**（M01、M02、M03……），每条素材给一个简短标题（≤15字，一句话概括）
2. 标注素材类型（`[证据]` / `[观点]` / `[行动]`，初步判断）
3. 若 references/ 里有文件，自动检查哪条素材可能关联到哪个文件

输出格式：

```
📋 素材清单（初稿，共 <N> 条）：

M01 [证据/观点/行动]  <素材标题，一句话>
M02 [证据]           <素材标题>
M03 [观点]           <素材标题>
...

📁 references/ 文件关联：
  references/monitor-screenshot.png → 可能与 M02 相关（数字证明）
  references/arch-diagram.png       → 可能与 M04 相关（架构说明）
  （无关联则不列）
```

然后问：

**Qd-1：素材清单看起来对吗？**

```
A. 基本对，继续分类
B. 漏了一些——我来补（再说几条，AI 追加到清单）
C. 有几条不准确——告诉 AI 哪条，怎么改
D. 某条应该删掉（告诉 AI 哪条）
E. B + C 都有（补充 + 修改）
```

> 💡 这个清单不需要完美，之后每步都可以追加。先有个雏形，我们再一起完善。

**等待用户确认后，AI 锁定素材清单，进入 Step 2。**

## Step 2：三类分类

AI 把素材清单按三类重新分组，展示分类结果：

```
📂 三类素材（分类结果）

── 证据类（案例/数据）──────────────────────────────
M01  <标题>   → 案例（自己经历的真实故事）
M03  <标题>   → 数据（监控数字/实验结论）
M07  <标题>   → 案例（较简短，可能需要补充）

── 思考与结论类（观点/方法论/洞察）────────────────
M02  <标题>   → 观点（反常识判断）
M05  <标题>   → 方法论（可复用做法）
M08  <标题>   → 洞察（Why 层待深化）

── 行动类（建议/checklist/下一步）────────────────
M04  <标题>   → 建议（给听众带走的动作）
M06  <标题>   → checklist（步骤型，可直接复用）

⚠️ 注意：M09 暂时无法分类，可能是开门/关门素材，单独列出
```

**Qd-2：分类结果准确吗？**

```
A. 准确，继续做质量深化
B. 有几条分错了（告诉 AI 哪条移到哪类）
C. 某条我想删掉（告诉 AI）
D. 还有几条没提到，我来补充
```

> 💡 行动类素材在很多演讲里容易被忽视——听众能带走"一个明天就能做的动作"，是演讲记忆度最高的设计之一。如果行动类素材少，后面可以专门补充。

**等待用户确认后，进入 Step 3。**

---

## Step 3：素材质量深化

> 逐条对每类素材做质量检查和追问。
> **原则：每条素材最多 1-2 个追问，不啰嗦，追问到位就推进。**

---

### 3-A：证据类深化

对每条证据类素材，AI 逐条检查：

**案例型素材检查：**

> 「M0X「<素材标题>」——这是你亲身经历的真实故事，还是行业通用案例？」

- 若是**行业通用案例**：
  > 「行业案例听众可能听过，效果打折。你工作中有没有类似的真实经历？哪怕是更小的版本？」
  > 💡 失败案例/踩坑案例往往比成功案例更有共鸣——听众会觉得"你不是在吹牛"

- 若是**真实案例**，AI 帮结构化为四段：

  **Qd-3：这个案例的四个段落，你现在能说出来吗？**
  ```
  背景（Context）：[项目/当时的处境/规模数字]
  冲突（Conflict）：[出现了什么问题？为什么难？谁受到影响？]
  解决（Resolution）：[你怎么分析？做了什么决策？为什么选这个方案？]
  结果（Result）：[量化数字 + 情景细节]
  ```
  > 💡 四段里「冲突」最重要——没有冲突的案例只是流水账，有冲突才有张力。不清楚的段落现在补，以后讲的时候就不会语塞。

**数据型素材检查：**

- 数字有没有来源（监控截图 / 调研报告 / 实验结果）？
- 听众已经知道这个数字吗？（已知 → 换角度，不已知 → 标注"有效证据"）
- 能不能和行业/历史数据做对比，让冲击力更强？

**汇报型专项：成果 vs 事情（仅汇报型触发）：**

> 「M0X 这条，描述的是**成果**还是**事情**？」

```
❌ 事情：建设质量体系
❌ 事情：上线了监控平台

✅ 成果：通过构建质量保障体系，XX 应用持续保持零故障单（12个月）
✅ 成果：监控平台上线后，故障定位时长从30分钟降至3分钟，Q3节约人力成本XX万
```

若是"事情"：引导讲者改成成果写法，加上"通过什么、达成了什么、带来什么影响"。

**汇报型专项：痛点数字化（仅汇报型触发）：**

> 「M0X 描述的痛点，能把它变成一个数字吗？」

```
从不痛到很痛的陈述梯度：
  不痛：我们是个 web 服务，需要提升 SLO
  一般：我们是个核心 web 服务，涉及业务关键流程
  较痛：web 服务涉及客户支付流程
  很痛：三个9的服务质量 = 每月损失超过1000万
```

**汇报型专项：成果截图（仅汇报型触发，针对核心成果数据）：**

**Qd-4：这个成果「<成果描述>」，有没有可以截图的证明？**

```
A. 有监控看板截图 / 数据大盘截图              ← 最有说服力
B. 有业务方或客户的认可（截图/文字/邮件）
C. A + B 都有（双重证明）                    ← 推荐（最可信）
D. 暂时没有，但可以去找
E. 这个成果不需要额外证明
```

> 💡 评委看到监控截图的那一刻，质疑会直接消失。10分钟找一张截图，比口头解释30分钟有效。

若选 A / B / C：引导讲者把文件保存到 `references/` 目录，并在素材里记录文件名。

---

### 3-B：思考与结论类深化

对每条观点/方法论类素材，AI 逐条检查：

**Why 三层推导检查：**

> 「M0X「<素材标题>」——你说了这个结论，但听众会在心里问：为什么？
> 能不能给我三层 Why？」

```
表层原因（听众容易想到的）：[...]
更深原因（一般人没想到的）：[...]
根因/方法论（你真正的洞察，可复用的）：[...]
```

> 💡 只有一层 Why 的内容，听众听完会说"对，知道了"；有三层 Why 的内容，听众会说"原来如此，我之前没想到这一层"——这才是演讲的价值。

**What/How vs Why 检查：**

```
听众通常知道：What（是什么）、How（怎么做）
听众通常不知道：Why（更深入的为什么）

❌ 只有 What/How：「我们用了 X 方案，分三步实施……」
✅ 有 Why：「为什么选 X 而不是 Y？因为我们发现……（Why 第一层）……根本原因是……（Why 第三层）」
```

**可复用性检查：**

> 「M0X 这个方法论，听众能用到自己的工作里吗？
> 如果他回去之后想复用这个方法，第一步应该做什么？」

💡 如果你说不出第一步，说明这个方法论还太抽象——需要再具体化。

**汇报型专项：技术递进（仅汇报型触发）：**

> 「行业里通常做到 A、B、C，你做了 D 甚至 E——
> 能不能描述一下 D 比 C 难在哪里？E 的突破点是什么？」

```
通常大家做到 A、B、C，
我们还做了 D（描述为什么 D 更难），
甚至做了 E（描述 E 的突破点）。
```

**汇报型专项：因果/运气检查（仅汇报型触发）：**

> 「M0X 这个成果，有没有因果推导？还是有点像"碰运气"？」

```
❌ 运气牛：我们在疫情期间做了这个，流量暴增，成功了
✅ 因果牛：疫情初期我判断流量会暴增，提前3周做了容量规划，
           具体依据是XX，最终顶住了X倍流量
```

> 「如果去掉外部因素（政策/市场风口），这个成果还成立吗？为什么？」

---

### 3-C：行动类深化

对每条行动/建议类素材，AI 逐条检查：

**可执行性检查：**

> 「M0X「<行动标题>」——听众听完之后，能"明天就做"吗？
> 还是太概念化，不知道第一步是什么？」

```
❌ 太空泛：建立观测体系
✅ 具体可执行：明天做一件事——找到你服务最慢的那个接口，看看它的 P99 耗时是多少。
```

如果太空泛：引导讲者给出第一步的具体操作。

---

### 3-D：假问题识别（全局扫描，所有类型）

AI 主动扫描所有素材，识别是否有"假问题"描述：

```
假问题的特征：用"不/没/非"描述的问题（实际上是在说解决方案的对立面，不是真正的问题）

❌ 假问题：K8S 的资源无法复用，导致无法支持更多业务
✅ 真问题：成本压力下，资源有限但业务增长快

❌ 假问题：缺乏统一的监控平台
✅ 真问题：故障发生时，平均定位时长超过30分钟，影响业务损失每小时XX万
```

若发现假问题，提示用户修正。

## Step 4：关联关系构建

> **这一步是 plan-details 最有价值的地方之一。**
> AI 帮你看清楚素材之间的关系——哪些是因果，哪些在时间上有先后，哪些相互支撑，哪些是同一类问题的不同层次。
> 这些关联关系是后面 /plan-mindmap 生成结构的核心依据。

AI 主动识别素材之间的关联关系，展示给讲者确认或补充：

**关联类型（金字塔原理三种关联 + 内在结构）：**

| 类型 | 说明 | 示例 |
|------|------|------|
| **因果关系** | A 导致 B，或 A 是 B 发生的原因 | `M03 是 M01 发生后的直接结果` |
| **时间关系** | A 在时间上发生在 B 之前 | `M01→M02→M03 依次发生` |
| **重要关系** | A 支撑 B，或 A 是整场演讲的核心 | `M06 是全场最重要的方法论` |
| **内在结构关系** | A 是 B 中描述的某个角色/类别/难题 | `M04 是 M06 描述问题中最难解决的那个` |

**展示格式（AI 生成，讲者确认）：**

```
🔗 素材关联关系图（AI 识别，请确认）：

因果关系：
  - M03 是 M01 发生后的直接结果（M01 是原因，M03 是结果）
  - M05 解释了 M02 为什么发生

时间关系：
  - M01 → M02 → M03 在时间上依次发生

重要关系：
  - M06 是全场最重要的方法论（其他素材都在为它做铺垫）
  - M02 和 M07 共同支撑 M06 的核心观点

内在结构关系：
  - M04 是 M06 描述的核心难题（M06 的方法论正是为解决 M04 而生）
  - M01 属于「四因说」中的动力因（触发了整个问题链条）
```

**Qd-5：关联关系准确吗？**

```
A. 准确，继续
B. 有几条不对——告诉 AI 哪条，怎么改
C. 还有 AI 没发现的关联——我来补充
D. 某条关联我觉得不对——告诉 AI 删掉哪条
E. B + C 都有
```

> 💡 关联关系不需要覆盖所有素材，只需要找到最重要的几条——演讲结构通常沿着最强的因果链或递进关系展开。

**等待用户确认后，AI 更新关联关系图，进入 Step 5。**

**文件格式：**

# Details — <演讲主题>

> 生成时间：<timestamp>
> 演讲类型：<分享型 | 汇报型>
> 目标时长：<X> 分钟
> 素材总数：<N> 条（证据 <a> 条 / 思考结论 <b> 条 / 行动 <c> 条）

---

## 素材库

### 证据类（<N> 条）

#### M01：<素材标题>
**类型：** 案例 / 数据

**内容：**
<具体内容>

（案例补充格式）

**冲突（Conflict）：** <出现了什么问题？为什么难？谁受到影响？>
**解决（Resolution）：** <你的判断、决策过程、为什么选这个方案>
**结果（Result）：** <量化数字 + 细节>

**来源/证明：** <references/文件名 or 暂无>
**质量标注：** ✅ 完整 / ⚠️ 待补充（<说明缺什么>）

### 思考与结论类（<N> 条）

## 附录：方法论速查

### 空姐效应
PPT 上写了的内容，演讲时就不要读了。听众眼睛盯着屏幕就离不开了。
只在 PPT 上放：关键词、数字、图示——不是你要说的完整句子。
→ 这一步整理的"演讲口头说"，就是 PPT 不会放的内容。

### 一波三折模板
```
挑战和失败：[产生共鸣的情景，听众在场的模样，自己的感受]
思考：[至少3层根因：原因 / 更深层原因 / 更更深层原因 + 推导逻辑 + 解决方案]
成功的结果：[量化数字 + 共鸣细节]
新挑战（递进，更难）：[同样有情景和细节]
再次思考：[与上次有递进关系，更深层更细节更本质]
成功的结果：[...]
最高潮挑战 → 最终成功：[...]
```

### F.A.B 法则
- **Feature（特性）**：你做了什么，技术实现是什么
- **Advantage（优势）**：相比现有方案，技术/工程上的优势在哪里
- **Benefit（价值）**：对业务、对用户、对团队的实际价值是什么

### 已知地图
听众通常知道：What（是什么）、How（怎么做）
听众通常不知道：**Why（更深入的为什么）**
→ 你的内容应该主要覆盖"Why"区域

### Why 三层推导示例
```
论点：「需要建立 SLO 体系」

第一层（表层，大家都能想到）：
  「没有 SLO，团队不知道服务好不好。」

第二层（更深，一般人没想到）：
  「更深的问题是：有了指标，但没有分级——P1 告警和 P3 告警用同一个处理流程，
   导致重要问题被淹没在噪音里，真正的 P1 响应时间还是很长。」

第三层（根因，可复用的方法论）：
  「根本原因是：指标、告警、处理流程三者没有联动。
   SLO 不只是监控指标，是一套「定义 → 接入 → 分级 → 复盘」的闭环机制。
   任何缺少这个闭环的团队，做再多告警也是救火，不是预防。」
```
