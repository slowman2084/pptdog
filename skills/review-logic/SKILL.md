## 审查维度

### 维度 1：推理链条类型检查

每个章节（mindmap 的 ## 节点）必须有一条清晰的推理链条支撑其论点。
AI 识别该章节用的是哪种推理方式，并检查是否执行到位：

| 推理类型 | 定义 | 识别信号 | 常见缺陷 |
|---------|------|---------|---------|
| **演绎推理** | 从普遍规律推出具体结论（大前提+小前提→结论） | 章节先立一个普遍原则，再代入具体案例得出结论 | 大前提没有建立，直接给结论；或小前提与大前提不匹配 |
| **归纳推理** | 从多个具体案例总结出规律 | 章节列举多个案例/数据点，最后归纳出共同规律 | 案例数量不足（只有1个）；案例之间不是真正的同类 |
| **溯因推理** | 从现象反推最可能的原因（Best Explanation） | 章节先呈现一个令人困惑的现象，然后给出最合理的解释 | 给出的解释不是"最可能的"，只是"可能的"；没有排除其他解释 |

**检查步骤：**
1. 识别每章用的推理类型（可能混用，但主线要清晰）
2. 检查推理链是否完整：
   - 演绎：大前提→小前提→结论，三段都有吗？
   - 归纳：案例够多够典型吗？归纳结论准确吗？
   - 溯因：现象描述清楚吗？解释的"最可能"依据是什么？
3. 检查各章节推理类型之间是否有冲突/矛盾

**severity 判定：**
- `critical`：章节没有可识别的推理链条（只是事实堆砌）
- `major`：推理链不完整（缺关键一环）
- `minor`：推理类型可以更明确

### 维度 2：MECE 检查

**检查逻辑：**
- 同一层级的论点（mindmap 的 ### 节点），是否属于同一维度？
- 是否有两个论点在说同一件事（有重叠）？
- 是否有明显遗漏的维度（某类情况没有覆盖）？

**生成 suggestion 格式：**
```
问题：论点 X.Y 和论点 X.Z 存在内容重叠
  「[论点 X.Y 标题]」提到 [重叠内容]
  「[论点 X.Z 标题]」也在说 [重叠内容]

Ql-mece-N：如何处理这两个论点？

A. 合并为一个论点：「[合并后标题]」
   → 保留 X.Y，将 X.Z 的内容合并进来
   → 预计减少 [N] 张 Slide

B. 明确区分：让 X.Y 专注 [某方面]，X.Z 专注 [某方面]
   → X.Y 修改为：[具体改法]
   → X.Z 修改为：[具体改法]

C. 保持原样（两个论点各有侧重，重叠是必要的铺垫）

> 💡 推荐 A，因为听众记不住两个相似的论点，合并后信息密度更高。
```

**severity 判定：**
- `major`：同层论点有明显重叠（超过 30% 内容相同）
- `minor`：MECE 有小缺口（有遗漏但影响不大）

### 维度 3：批判性思维——隐含假设检查

**什么是隐含假设：**
演讲中很多论点依赖一个"说话人认为理所当然但听众未必认可"的前提。
如果这个假设不成立，整个论点就垮了。

**AI 识别逻辑：**
对每个核心论点（尤其是结论性论点），追问：
> "这个论点成立，需要依赖哪些我没有明说的假设？"

常见隐含假设类型：
- 因果假设："因为 A，所以 B"——真的是 A 导致了 B 吗？有没有第三变量？
- 普遍化假设："我们团队这样，所以大家都这样"——样本够代表吗？
- 优先级假设："这比那重要"——听众认可这个优先级吗？
- 可行性假设："这在你们那里也能做"——听众的情况和讲者真的一样吗？

**生成 suggestion 格式：**
```
问题：论点「[论点标题]」有一个隐含假设可能需要显式建立

隐含假设：[描述这个假设]
风险：如果听众不认可这个假设，这个论点就难以共鸣

Ql-assumption-N：如何处理这个隐含假设？

A. 在论点开头显式建立这个假设
   → 在演讲口头说加一句：「[具体内容，例：相信在座的很多团队也面临同样的压力——]」

B. 用数据/互动让听众自己确认假设
   → 在演讲口头说加一句：「[具体互动话术，例：有多少人的系统里现在还有超过5年的老代码？]」

C. 保持原样（假设已经足够隐蔽，不需要处理）

> 💡 推荐 A，显式建立假设可以防止听众在心里默默反驳。
```

**severity 判定：**
- `major`：核心结论依赖一个高风险隐含假设（听众可能明确不认可）
- `minor`：假设存在但风险较低

### 维度 4：整体金字塔结构完整性（全局视角）

> 补充 review-structure 的局部开关门检查，从整体俯瞰金字塔。

**检查逻辑：**
1. **顶层结论**是否在开头就清晰呈现？（汇报型：结论先行；分享型：开头钩子）
2. **各章节论点**加在一起，是否能支撑顶层结论？（支柱完整性）
3. **子论点**是否都在支撑其父章节，没有游离的论点？

**生成 suggestion 格式：**
```
问题：第N章论点「[标题]」与整体顶层结论「[顶层结论]」的支撑关系不清晰

Ql-pyramid-N：如何建立这章与顶层结论的连接？

A. 在本章小开门加一句桥接：「[具体桥接话术]」
B. 调整章节顺序，让支撑关系更明显：[具体建议]
C. 这章其实是背景/铺垫，调整为"背景章节"定位（不需要直接支撑结论）

> 💡 推荐 A，听众在听每章时需要知道"这章在整体中处于什么位置"。
```

**severity 判定：**
- `major`：某章论点与顶层结论的关系不明确
- `minor`：关系存在但桥接不够显式

## JSON 生成规则

### 建议数量约束

> **AI 必须严格遵守：**
> - 每次审查**最少生成 5 条**建议（critical + major + minor 合计）
> - 若某个维度审查后无问题，才允许该维度为 0——但其他维度必须补充，总数 ≥ 5
> - 不允许因为"逻辑看起来还不错"而少于 5 条——逻辑永远有可以追问的地方
>
> **自查提示（输出前 AI 必须做）：**
> - 每个章节都至少追问了一次"这个论点依赖什么隐含假设"吗？
> - MECE 检查是从 mindmap 的层级结构出发逐层检查的吗？
> - 整体俯瞰：顶层结论是否被所有章节合力支撑了？


> **context 和 problem 字段的写法要求（AI 必须遵守）：**
> - `context` 用一句话说清楚位置：`"第N章·论点X.Y「标题」——[具体位置]"`，让讲者不翻文件就能定位
> - `problem` 必须说人话，禁止用方法论术语堆砌：
>   - ❌ 「隐含假设违规，因果推导链存在逻辑断层」
>   - ✅ 「你说了结论，但没说依据——听众会在心里问：凭什么？」
> - `options` 里每个选项的 `content` 必须是可以直接用的文字，不是方向描述

AI 完成所有维度的审查后，生成以下格式的 JSON，写入 `logic-<ts>.json`：

```json
{
  "reviewer": "logic",
  "reviewer_id": "logic",
  "slug": "<slug>",
  "ts": "<ISO timestamp>",
  "suggestions": [
    {
      "id": "logic-001",
      "reviewer_id": "logic",
      "dimension": "推理链条 / MECE / 隐含假设 / 金字塔结构",
      "severity": "critical | major | minor",
      "source_lines": { "start": 10, "end": 20 },
      "original": "原始内容片段",
      "context": "第N章·论点X.Y「[论点标题]」——[问题发生在哪个部分]",
      "problem": "用一句人话描述问题（不用技术术语，让不了解方法论的人也能看懂）",
      "options": [
        { "label": "A", "content": "具体改法（可直接复制粘贴的内容）" },
        { "label": "B", "content": "具体改法" },
        { "label": "C", "content": "保持原样" }
      ],
      "recommended": "A",
      "recommended_reason": "推荐 A 的理由"
    }
  ]
}
```

> **options 质量要求：**
> - 每个 option 的 content 必须是**可以直接复制粘贴**的具体文字或操作
> - 不允许写「在此处添加相关内容」「进一步说明论点」这类方向性描述
