# 理想态：review-logic

## 概述

`review-logic` 是 pptdog 审查链路中专注于**逻辑完整性**的审查器。它审查演讲的推理质量——每个章节的推理链是否成立、同层论点是否 MECE、论点是否依赖未建立的假设、所有章节是否合力支撑顶层结论。

与 `review-structure` 检查叙事弧度不同，`review-logic` 关注的是**论点本身的逻辑严密性**：听众质疑"凭什么"时，演讲能不能经得住追问。

---

## 需求体系

### 固有需求分类（视角 A：逻辑缺陷类型）

| ID | 固有需求 | 定义 |
|----|---------|------|
| I1 | 推理完整性 | 每章推理链条（演绎/归纳/溯因）是否成立，关键一环是否缺失 |
| I2 | 结构无冗余 | 同层论点无重叠、无明显遗漏（MECE） |
| I3 | 假设可见性 | 论点依赖的隐含假设被识别出来，让讲者决定是否显式建立 |
| I4 | 全局支撑力 | 所有章节论点合力支撑顶层结论，无游离论点 |

### 主需与次需

**主要需求：**

- **I1 推理完整性** → 落地为：每章强制识别主线推理类型（演绎/归纳/溯因），即使混杂也找出主线（D1A）；主线类型确定后，检查对应推理链的完整性（三段论缺哪段 / 归纳案例数不足 / 溯因未排除竞争解释）
- **I3 假设可见性** → 落地为：对所有核心论点追问隐含假设；所有未在演讲中显式建立的假设都生成 suggestion（D3B），让讲者自己决定哪些要处理；四类假设（因果/普遍化/优先级/可行性）全部在扫描范围内

**次要需求：**

- **I2 结构无冗余** → 两层都查（D2B）：章节间（## 节点）和章节内论点（### 节点）均检查重叠/遗漏；chapter 层重叠为 major，论点层重叠为 minor（通常）
- **I4 全局支撑力** → 从整体俯瞰：每章与顶层结论的支撑关系是否清晰，是否有游离于顶层结论之外的章节

```json
{
  "reviewer": "logic",
  "reviewer_id": "logic",
  "slug": "<slug>",
  "ts": "<ISO timestamp>",
  "suggestions": [
    {
      "id": "logic-001",
      "reviewer_id": "logic",
      "dimension": "推理链条 | MECE | 隐含假设 | 金字塔结构",
      "severity": "critical | major | minor",
      "context": "第N章·论点X.Y「标题」——问题所在部分",
      "source_lines": { "start": N, "end": M },
      "original": "从 slide-content.md 直接引用的原文片段",
      "problem": "口语化描述逻辑问题（不用方法论术语）",
      "options": [
        { "label": "A", "content": "具体可直接使用的修改文字", "recommended": true, "recommended_reason": "推荐理由" },
        { "label": "B", "content": "...", "recommended": false },
        { "label": "C", "content": "保持原样", "recommended": false }
      ]
    }
  ]
}
```

**options.content 写法规范**：
- 推理链条：写出补全关键一环的具体文字（补大前提/补案例/补排除竞争解释的话术）
- MECE：写出合并后的标题，或区分两个论点各自专注点的具体改法
- 隐含假设：写出显式建立假设的具体话术（"相信在座很多团队也..."）
- 金字塔：写出桥接顶层结论的小开门话术
- **禁止**：方向说明（"建议补充依据"/"需要添加假设建立"等）

## 质量标准

### Preamble 执行

- 版本检查 + learnings 搜索 + slug 识别
- 读取 slide-content.md（不存在→停止）+ mindmap.md + details.md（可选）
- 打印文件统计（行数/章节数）

### 维度 1：推理链条类型（D1A — 强制识别主线）

**识别流程**：
1. 对每个章节（mindmap 的 ## 节点对应的 slide-content 章节），识别其推理方式的**主线类型**：
   - **演绎**：先立普遍原则，再代入具体案例得出结论（大前提→小前提→结论）
   - **归纳**：列举多个案例/数据点，归纳出共同规律
   - **溯因**：先呈现令人困惑的现象，给出最可能的解释
2. 若章节混用多种推理，**选取占比最大的为主线类型**，在 context 字段注明"主线：溯因，混有部分归纳"
3. 判断主线推理链是否完整：

| 主线类型 | 完整性检查 | 触发 suggestion 的信号 |
|---------|-----------|----------------------|
| 演绎 | 大前提/小前提/结论三段都有 | 缺大前提（直接给结论）；或小前提与大前提不匹配 |
| 归纳 | 案例数量 ≥2 且案例间是真正同类 | 只有1个案例就下结论；案例异质（不是真正同类）|
| 溯因 | 现象清晰 + 解释是"最可能"的（排除了竞争解释）| 给出的解释只是"可能"而非"最可能"；未排除其他明显解释 |

severity：`critical`=章节无可识别的推理链（纯事实堆砌，无论证逻辑） / `major`=推理链不完整（缺关键一环）/ `minor`=推理类型可以更明确

**推理类型输出规范**：每章审查结果在 context 字段注明识别到的推理类型，即使无问题也在摘要中列出（帮助讲者了解自己的论证风格）。

### 维度 2：MECE 检查（D2B — 章节层 + 论点层两层）

**两层检查**：

**层 1：章节间（## 节点）**
- 各章节是否覆盖了支撑顶层结论所需的完整维度？
- 是否有两个章节在说实质上相同的事？
- 章节层重叠 = major；章节层遗漏（有明显缺口）= major

**层 2：章节内论点（### 节点）**
- 同一章节内的并列论点是否属于同一分析维度？
- 是否有两个论点内容超过 30% 相同？
- 论点层重叠 = minor（通常）；论点层遗漏（某类情况明显未覆盖）= minor

**options 写法**：
- 重叠：写出合并后的具体标题，和保留哪个/删除哪个的操作建议
- 遗漏：写出缺失维度的具体论点标题，附说明这个维度为什么重要

severity：`major`=章节层重叠/遗漏 / `minor`=论点层重叠/遗漏

### 维度 3：隐含假设（D3B — 所有非显式假设）

**识别流程**：
1. 对每个核心论点（不含纯背景/过渡页），追问：
   > "这个论点成立，需要依赖哪些演讲中没有明说的假设？"
2. 识别四类假设：
   - **因果假设**："A 导致 B"——有没有第三变量？
   - **普遍化假设**："我们这样，所以大家都这样"——样本够代表吗？
   - **优先级假设**："这比那重要"——听众认可这个优先级吗？
   - **可行性假设**："这在你们那里也能做"——听众情况和讲者一样吗？
3. **只要假设未在演讲中被显式建立，就生成 suggestion**（D3B）
   - 共识性假设（所有听众都认可的常识，如"工程质量影响业务稳定性"）可豁免
   - 豁免理由需在 context 字段注明

**options 写法**：
- A：在论点开头显式建立假设的具体话术（"相信在座很多团队也..."/"有多少人遇到过..."）
- B：用数据/互动让听众自己确认假设的具体话术
- C：保持原样（认为假设是共识）

severity：`major`=核心结论依赖高风险假设（听众明确可能不认可） / `minor`=假设存在但风险较低（多数听众可能认可）

### 维度 4：金字塔整体结构

**检查逻辑**（补充 review-structure 的局部检查，从整体俯瞰）：
1. **顶层结论**：开篇是否有清晰的顶层结论？（汇报型：结论先行；分享型：核心观点可从关门推出）
2. **各章节合力支撑**：每章论点是否都在支撑顶层结论？有没有"游离章节"（与结论无直接关系）
3. **子论点归属**：子论点是否都在支撑其父章节，而不是跳层支撑顶层结论

**options 写法**：
- 游离章节：写出桥接话术（在该章小开门加"这章支撑顶层结论的哪个维度"的说明）
- 结论不清晰：写出顶层结论的具体表达（一句话版本）

severity：`major`=某章论点与顶层结论的关系完全不清晰 / `minor`=关系存在但桥接不够显式

### 建议数量约束

- 期望范围：5-12 条
- 最低保证：≥5 条（若真的很少，在摘要说明原因）
- 自查：隐含假设是最容易遗漏的——生成前检查是否每章核心论点都追问过假设

## 失败模式

### F1：推理类型无法识别时不生成建议（严重）

**特征**：某章节内容是纯事实列举（"我们在Q1做了A，Q2做了B，Q3做了C"），AI 未生成任何推理链条 suggestion，直接跳过。

**危害**：事实堆砌正是最严重的推理缺陷（severity=critical），跳过等于让最严重的问题漏网。无论如何都要报——"这章没有论证逻辑，只有事实罗列"本身就是 critical。

**触发条件检测**：在纯事实堆砌的章节测试用例中，检查是否生成了 dimension=`推理链条`、severity=`critical` 的 suggestion。

### F2：隐含假设被跳过（常见）

**特征**：某论点依赖"听众和讲者面临同样的技术栈复杂度"这一可行性假设，AI 未生成任何假设 suggestion。

**危害**：隐含假设是讲者最难自己发现的盲点，也是听众"感觉对不上"的根源。大量隐含假设漏报会导致讲者不理解为什么听众反应冷淡。

**触发条件检测**：对每个核心论点检查 suggestions 中是否有至少1条 dimension=`隐含假设` 的建议（允许存在0条仅当假设已显式建立）。

### F3：MECE 检查只查论点层，跳过章节层

**特征**：review-logic 检查了章节内论点的重叠，但没有检查两个章节之间是否有实质重叠（D2B 决策要求两层都查）。

**危害**：章节层重叠是更严重的结构问题——两个章节说同一件事会让听众困惑"为什么分成两章"，比论点层重叠影响更大。

**触发条件检测**：在章节间存在明显重叠的测试用例中，检查是否有 dimension=`MECE`、context 指向章节层的 suggestion。

### F4：Options 只给方向说明

**特征**：推理链条 suggestion 的 options 写"建议在开头补充大前提"，而不是写出具体的大前提文字。隐含假设的 options 写"建议在此处建立假设"，而不是写出"相信在座很多团队也面临..."这样的具体话术。

**危害**：讲者看完不知道大前提该怎么说、假设该怎么建立，建议无法落地。

**触发条件检测**：options content 字段是否包含具体话术文字（直接可粘贴），而非"建议/应该/需要"等方向词。

### F5：对背景/过渡页做隐含假设检查

**特征**：演讲中纯背景铺垫页（"让我先介绍一下我们团队的背景"）被做了隐含假设检查，生成了无意义的建议。

**危害**：背景页的目的是信息传递，不是论证——对其追问"因果假设"会产生无意义的 suggestion，降低信噪比。

**触发条件检测**：对明确标注为背景/铺垫的 Slide，检查是否有 dimension=`隐含假设` 的 suggestion（应为0条）。

## 评分维度

### D1：推理链条识别与完整性检查（25 分）

关联主需 **I1 推理完整性**。

**量化检测**：
- 每章正确识别主线推理类型（演绎/归纳/溯因）→ 各 +3 分（上限 15 分）
- 有断链的章节被正确标记 → 各 +5 分
- 纯事实堆砌章节生成 critical suggestion → 必要条件

| 分数段 | 描述 |
|--------|------|
| 22-25 | 每章主线推理类型正确识别（即使混杂也确定主线）；推理链断裂准确标记；纯事实堆砌判为 critical；options 写出补全关键一环的具体话术 |
| 15-21 | 大多数章节类型识别正确，但有 1-2 章推理断裂漏报；或 options 缺乏具体补全文字 |
| 8-14 | 超过半数章节推理类型识别有误；或纯事实堆砌章节被跳过 |
| 0-7 | 未执行推理类型识别；或所有章节均判为"无问题" |

### D2：隐含假设覆盖率与话术质量（25 分）

关联主需 **I3 假设可见性**。

**量化检测**：
- 每个核心论点至少追问一次假设 → 核心论点覆盖率 × 25 分
- options 写出具体建立假设的话术 → 必要条件
- 共识性假设豁免有说明 → +3 分加成

| 分数段 | 描述 |
|--------|------|
| 22-25 | 所有核心论点均被追问假设；所有非显式、非共识假设生成 suggestion；options 写出具体建立假设的话术（"相信在座..."/"有多少人..."）；豁免假设有说明 |
| 15-21 | 大多数核心论点被检查，但遗漏 1-2 处；或 options 话术不够具体 |
| 8-14 | 只检查了部分论点（如只检查最后一章）；或大量假设被豁免但无说明 |
| 0-7 | 未执行隐含假设检查；或生成了完全无意义的假设 suggestion（针对背景页）|

### D3：MECE 两层完整性（20 分）

关联次需 **I2 结构无冗余**。

**量化检测**：
- 章节层（## 节点间）MECE 检查 → 10 分
- 论点层（### 节点间）MECE 检查 → 10 分

| 分数段 | 描述 |
|--------|------|
| 17-20 | 章节间和论点间均检查重叠/遗漏；章节层重叠标为 major，论点层标为 minor；options 写出合并标题或区分专注点的具体改法 |
| 12-16 | 执行了两层检查，但遗漏或重叠的判断有 1-2 处不准确 |
| 6-11 | 只执行了论点层检查，未检查章节层 |
| 0-5 | 未执行 MECE 检查 |

### D4：金字塔全局检查（15 分）

关联次需 **I4 全局支撑力**。

| 分数段 | 描述 |
|--------|------|
| 13-15 | 正确识别顶层结论；检查每章与顶层结论的支撑关系；游离章节（与结论无关）标为 major；options 写出具体桥接话术 |
| 9-12 | 识别了顶层结论，但游离章节漏报；或 options 只说方向不给话术 |
| 0-8 | 未执行金字塔全局检查；或把所有章节都判为"与结论相关"（无区分能力）|

### D5：格式合规与建议数量（15 分）

**量化检测**（4 项，各约 4 分）：

| # | 检测项 | 通过标准 |
|---|--------|---------|
| 1 | 建议总数 ≥ 5 条 | 不足时需说明原因 |
| 2 | source_lines 真实行号 | 不可估计，需从文件读取 |
| 3 | problem 字段无方法论术语 | 不含"MECE 违规/推理链断裂/隐含假设违规"等 |
| 4 | JSON 格式合规 | schema 通过，所有必填字段存在 |
