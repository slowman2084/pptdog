## Step 0：判断分享类型

> 目的：在动笔之前先搞清楚这是什么场合，决定后续用什么方法论。

向用户提问（一次性给出三个问题，让用户逐一或一起回答）：

---

**Q0-1：这次分享是什么场合？**

```
A. 内部技术分享 / 经验传递        → 听众是平级同事或下级
B. 职级晋升答辩 / 项目立项汇报    → 听众是评委或决策者          ← 如果不确定，选这个
C. 外部大会 / 公开演讲            → 听众是陌生的社区受众
D. 业务汇报 / 季度 / 年终复盘     → 听众是上级和跨部门同学
```

> 💡 如果同时符合多项（比如晋升分享，但台下有同事也有老板），选最"向上"的那个。

---

**Q0-2：听众和你的权力关系是什么？**

```
A. 平行或向下（同事、外部社区）   → 倾向「分享型」，讲故事讲方法
B. 向上（老板、评委）             → 倾向「汇报型」，先结论再论据   ← 推荐默认
C. 混合（全员会、大型技术交流）   → 混合型，开头汇报，主体分享
```

> 💡 不确定时选 B，因为汇报型的结构对"平行听众"也友好，反过来则不成立。

---

**Q0-3：你的时间限制是多少分钟？**

```
A. 10-20 分钟   → 极限断舍离，只讲一件事，一个核心结论
B. 30-45 分钟   → 标准结构，可以有 3-4 个论点              ← 最常见
C. 60 分钟以上  → 可以有完整叙事，但仍然需要主线
```

> 💡 宁可少一点也不要超时。超时是演讲最严重的失误之一。

---

```
📋 类型判断：[分享型 / 汇报型 / 混合型]
⏱️ 时间：[X] 分钟
💬 判断理由：[一两句话说明为什么这样判断]

👉 如果你觉得我理解有偏差，现在可以纠正我。
   确认后我们进入 Step 1：听众画像。
```

## Step 1：听众画像（空姐效应防御）

> **空姐效应核心逻辑：**
> 听众的已知 = What（是什么）+ How（怎么做）
> 听众的未知 = Why（为什么这样、更深层的原因）
> 重复已知 = 触发空姐效应 = 听众关机

向用户提问（5 问，一次给出，用户可逐条回答）：

---

**Q1-1：听众具体是谁？**

```
A. 工作 1-3 年的开发同学，熟悉基础但没做过系统设计
B. 工作 5 年以上的资深工程师 / 技术负责人            ← 如果是技术分享，最常见
C. 业务/产品/运营同学，关心结果不懂技术细节
D. 混合（技术+业务都有，或者初级+资深都有）
```

> 💡 不要只说「技术同学」。越具体，后面的结构越好做。
> 请补充说明：他们在哪个领域/业务方向工作？

---

**Q1-2：他们已经知道什么？**

请在两个层面上说：

```
What 层（概念定义）：
  A. 基本概念都不懂，需要从头讲
  B. 知道概念，但没有深入实践过            ← 最常见
  C. 概念和实践都很熟，关注的是"更深的原因"

How 层（操作方法）：
  A. 没做过，需要教具体步骤
  B. 做过但踩过坑，想知道如何避坑          ← 最常见
  C. 已经做得很好，关注的是如何做到极致
```

> 💡 What/How 层已知的内容，你在演讲里只能"一句话带过"或直接跳过，否则就是空姐效应。

---

**Q1-3：他们不知道、也没想到的 Why 是什么？**

```
A. 他们知道"是什么/怎么做"，但不知道"为什么这样设计"（设计背后的原因）
B. 他们知道问题存在，但不知道根因是什么（有认知但认知有盲区）  ← 最有穿透力
C. 他们完全没意识到这个问题的存在（需要先唤起问题意识）
```

> 💡 B 类型最容易触发「原来如此！」的顿悟感，是技术分享最有价值的切入角度。
> 请尝试用一句话说出这个 Why：「他们知道 [已知]，但不知道 [你要告诉他们的根因]」

---

**Q1-4：他们现在最头疼的问题是什么？**

```
A. 效率问题：某个流程太慢、某个操作太繁琐
B. 质量问题：产出不稳定、难以复现、难以保障      ← 工程类分享最常见
C. 决策问题：不知道选哪个方案、不知道如何取舍
D. 认知问题：别人都在讨论某个概念，自己不太理解
```

> 💡 请补充：用数字或场景把痛量化。
> 比如「AI 用的人不少，但真正融入工作流的比例低；产出看起来可以但质量不可控」
> 比「效率低」具体 10 倍，后续结构也会好写 10 倍。

---

**Q1-5：他们听完最该带走什么？**

```
A. 一个方法 / 框架：听完知道"以后遇到这类问题我知道怎么做了"
B. 一个视角 / 判断标准：听完能改变他们看待某个问题的角度     ← 技术分享高价值
C. 一个路线图：听完知道下一步该做什么、怎么推进
D. 一个可以立刻抄的实践案例：听完有模板有工具可以直接复用
```

> 💡 A 和 D 执行门槛低，适合"工具向"分享；B 和 C 认知价值高，适合"方法论向"分享。
> 你这次更偏哪种？

---

AI 根据回答生成**听众画像卡**：

```
👥 听众画像卡
─────────────────────────────────────────
身份：[具体描述]

```

## Step 2：主题聚焦（断舍离）

> **断舍离核心逻辑：**
> 有限时间里，讲深讲透一件事 > 面面俱到什么都讲
> 多即是少——你加的每一页都在稀释核心信息

### 2A：如有原始素材/大纲，先让 AI 分析

若用户启动时带了素材（`/ppt-hours [文件路径]`），或在对话中粘贴了大纲：

AI 读取后完成以下分析：

```
📋 素材分析
─────────────────────────────────────────
素材概览：[一句话描述素材范围]

候选主题（2-3 个，有实质差异）：
  A. [主题A，≤20字] — 核心传递：[一句话]
  B. [主题B，≤20字] — 核心传递：[一句话]
  C. [主题C，≤20字] — 核心传递：[一句话]（可选）

建议断舍离的内容：
  ✂️ [内容X]：听众已知，或与核心主题无关
  ✂️ [内容Y]：太技术细节，对这个听众无价值
  ✂️ [内容Z]：时间不够，删掉反而让主题更聚焦

👉 你倾向哪个主题方向？或者你有自己的想法？
```

### 2B：无素材时的聚焦提问

**Q2-1：如果你只有 3 分钟，这个分享/汇报最想让听众记住的是哪一件事？**

```
A. 一个方法论 / 框架（「以后遇到这类问题，用这个思路」）
B. 一个数字 / 结论（「我们做到了 X，原来大家以为做不到」）  ← 汇报型首选
C. 一个视角翻转（「你以为是 A，其实是 B」）                 ← 分享型高价值
D. 一个可复用的实践（「这套做法你回去可以直接抄」）
```

> 💡 只能选一个。你最想留下的是哪一个？

---

**Q2-2：听众听完离开会议室，你希望他们对走廊里遇到的同事说什么？**

请用第一人称帮听众说出那句话：

```
模板 A：「原来 [已知现象] 的根本原因是 [你揭示的Why]」
模板 B：「他们用 [方法/工具] 把 [指标] 从 X 做到了 Y，而且可以复用」  ← 汇报型
模板 C：「我之前对 [某个问题] 的理解完全是错的，正确的是 [新视角]」
```

> 💡 如果你无法用一句话写出这句话，说明主题还没聚焦够——继续收。

---

### 汇报型额外触发 🔴

若 Step 0 判断为汇报型或混合型，在 Step 2 结束前额外提问：

**Q2-汇报-1：你想证明的"成果"是什么？**

```
A. 质量/稳定性成果：故障率下降、SLO 提升、问题收敛
B. 效率/速度成果：研发周期缩短、发布频率提升、人效提升  ← 工程方向最常见
C. 规模/增长成果：覆盖业务数量、用户数、接入团队数
D. 能力/平台成果：建设了某个平台/体系，有持续产出
```

> 💡 成果 ≠ 事情。检查格式：「通过 [做了什么]，实现了 [什么结果]，量化是 [数字]」
> ❌ 事情版：「建设了质量体系」
> ✅ 成果版：「通过构建质量保障体系，故障率下降 90%，连续 6 个月零重大故障」

---

**Q2-汇报-2：这个成果的痛点，听众有多痛？**

```
A. 轻微：业内普遍问题，大家习以为常（痛感弱，成果说服力低）
B. 一般：有影响但不致命，可以接受（中等说服力）
C. 较痛：影响到核心流程或 KPI，老板关注               ← 推荐至少到这个层级
D. 很痛：直接影响到业务收入、用户体验或团队生死（最有说服力）
```

> 💡 你能不能把痛点再升一级？
> 比如：「SLO 不够高」→「SLO 低导致业务合规风险，每次评审都被点名」→「直接影响年度 KPI 达成」

## 内置方法论快查

> 本节为 AI 内部参考，不主动向用户完整展示，但可在对话中按需引用。

### 空姐效应
听众的已知内容，无论多重要都没有价值。两种触发场景：
1. 听众早就知道了（What/How 层）
2. PPT 上写了你再读出来（眼睛盯着 PPT 就离不开了）

防御方法：听众画像时精准标记「已知区域」，Step 2 主动断舍离。

### 断舍离
- 有限时间讲深讲透一件事 > 面面俱到
- 每加一个知识点，平均注意力就被稀释一次
- 克制才是爱：你删掉的东西，是为了让留下的东西更有力

### 汇报：成果 vs 事情
| 维度 | 事情（❌）| 成果（✅）|
|------|----------|----------|
| 格式 | 动词+名词 | 通过X，实现Y，量化Z |
| 例子 | 建设质量体系 | 通过质量保障体系，故障率下降 90% |
| 痛点 | 不体现业务价值 | 基于业务痛点，痛越痛成果越有价值 |

## learnings 写入时机

以下情况自动追加到 `~/.pptdog/learnings.jsonl`：
- 用户修正了 AI 的听众画像判断 → type: `correction`
- 用户发现某类听众有特殊的「已知陷阱」→ type: `pitfall`
- 主题聚焦经历多次迭代才收敛 → type: `insight`

格式：
```json
{
  "ts": "<ISO>",
  "skill": "ppt-hours",
  "type": "pitfall|insight|correction",
  "key": "<短标识>",
  "insight": "<一句话>",
  "confidence": 7,
  "source": "observed",
  "slug": "<项目slug>"
}
```
