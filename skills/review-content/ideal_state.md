# 理想态：review-content

## 概述

`review-content` 是 pptdog 内容审查链路的专项审查器，由 `/ppt-review` 自动调用，负责对 `slide-content.md` 进行内容质量的六维度审查：空姐效应、论点深度（Why层）、真实案例完整性、假问题识别、授人以渔程度、数字/证据充分性。

这个 Skill 的核心定位是**建立讲者对审查系统的信任**。建立信任的唯一方式是：报的问题真的有问题，提的修改真的能用。**误报（把好内容标成问题）比漏报更危险**——它让讲者开始怀疑所有建议的价值，最终全部忽略，包括真正重要的那些。

---

## 需求体系

### 固有需求分类（视角 B：讲者信任建立）

| ID | 固有需求 | 定义 |
|----|---------|------|
| I1 | 信号精准 | 只报真正影响听众体验的问题，不产生噪音 |
| I2 | 方案具体 | 每条建议后面有能直接用的修改文字，不是方向说明 |
| I3 | 理由可信 | problem 字段用口语描述真实后果，不说方法论术语 |
| I4 | 分级诚实 | critical 真的很严重，不滥用；minor 代表小事 |
| I5 | 机器可读 | JSON schema 合规，source_lines 真实行号，Dashboard 能渲染 |

### 主需与次需

**主要需求（决定审查结果的核心质量）：**

- **I1 信号精准** → 落地为：平衡门槛（D1B）——明确存在的问题必须报，minor 遵循"这里有真实改进空间"标准，不确定的不报；AI 推断内容跳过内容审查（D2A），单独一条提示；同类问题逐页报告（D3A），每页独立处理
- **I2 方案具体** → 落地为：options 的 content 字段必须写出能直接采用的修改文字（不是"建议精简 PPT 内容"这类方向说明）；problem 字段写出真实后果（"听众会去读屏幕不听你讲"，不是"空姐效应违规"）

**次要需求（影响建议质量的细节）：**

- **I3 理由可信** → problem/reason 字段用讲者能感同身受的语言，描述问题的真实后果
- **I4 分级诚实** → critical 只给真正影响演讲可信度的核心问题；major 给重要但不致命的问题；minor 给有改进空间但不影响大局的问题
- **I5 机器可读** → JSON 数组格式、所有必填字段完整、source_lines 从文件实际读取

### 需求间关系

```
I1(信号精准) ──── 基础：没有信号精准，讲者会忽略所有建议
I2(方案具体) ──── 价值：精准信号+具体方案=讲者立刻可以行动

I3(理由可信) ←── I1/I2 的表达质量：精准建议还需要用让讲者共鸣的语言说出来
I4(分级诚实) ←── I1 的优先级维度：精准信号还需要告诉讲者该先改哪个
I5(机器可读) ←── 系统层需求：精准建议还需要能被 Dashboard 正确渲染
```

## 输出规格

### JSON 输出文件

输出路径：`~/.pptdog/projects/<slug>/review-suggestions/content-<ts>.json`

JSON 数组，每条 suggestion 结构：

```json
{
  "id": "content-001",
  "reviewer_id": "content",
  "dimension": "空姐效应",
  "severity": "critical | major | minor",
  "context": "第N章·论点X.Y「论点标题」——问题所在部分",
  "source_lines": { "start": N, "end": M },
  "original": "从 slide-content.md 直接引用的原文片段",
  "problem": "口语描述问题真实后果，不用方法论术语",
  "options": [
    {
      "label": "A",
      "description": "改法简述",
      "content": "具体修改后的文字（必须是能直接用的内容）",
      "recommended": true,
      "reason": "为什么推荐这个改法"
    },
    {
      "label": "B",
      "description": "...",
      "content": "...",
      "recommended": false,
      "reason": "..."
    },
    {
      "label": "C",
      "description": "保持原样",
      "content": "（不修改）",
      "recommended": false,
      "reason": ""
    }
  ]
}
```

**字段规范：**

| 字段 | 类型 | 必填 | 规范 |
|------|------|------|------|
| id | string | ✅ | `content-NNN`，三位序号，按发现顺序递增 |
| reviewer_id | string | ✅ | 固定 `"content"` |
| dimension | string | ✅ | 枚举：`空姐效应 / 论点深度 / 真实案例 / Why层 / 假问题 / 授人以渔 / 数字证据 / AI推断内容` |
| severity | string | ✅ | `critical / major / minor` |
| context | string | ✅ | `"第N章·论点X.Y「标题」——问题所在部分"` |
| source_lines | object | ✅ | 从文件实际读取的真实行号，不可估计 |
| original | string | ✅ | 从 slide-content.md 直接引用 |
| problem | string | ✅ | 口语化，描述真实后果，禁止方法论术语 |
| options | array | ✅ | 3-5 个选项，content 必须是可直接采用的文字 |

**关于 options.content 的写法：**
- 文字改写：直接写改写后的文字
- 布局调整：写出布局名 + 布局具体内容
- 保持原样：写 `"（不修改）"`
- **禁止写法**：`"建议精简 PPT 内容"`、`"可以增加具体数字"`、`"建议改进表达方式"`

### 建议数量约束（平衡门槛）

- **合理范围**：内容 ≤ 20 页的文件期望 5-10 条；> 20 页期望 8-15 条
- **最低保证**：总数 ≥ 5 条（若某文件质量极高，允许低于保底，但 AI 应在摘要中说明原因）
- **不强求凑数**：宁可只报 3 条真实问题，也不报 10 条掺杂噪音的建议
- **自查要求**：生成前数一数总数，若 < 5 条，回头重扫每个维度确认是否真的质量极高

```

  details.md       : N 行
  mindmap.md       : N 行

🔍 正在审查内容质量（6个维度）...
```

## 质量标准

### Preamble 执行

- 运行版本检查 + learnings 搜索
- slug 识别（0个→停止 / 1个→自动选 / 多个→列出 / 传入→直接用）
- 读取 slide-content.md（不存在时立即停止，提示先运行 /slide-content-and-scripts）
- 打印文件统计（slide-content.md 行数/页数 / details.md 行数 / mindmap.md 行数）

### AI 推断/需补充内容识别（D2A）

**扫描流程**：
1. 读取 slide-content.md 时，识别所有含 `[AI推断，请确认]` 或 `[需补充：` 标注的内容块
2. **这些内容块不做六维度内容质量审查**（不对推断的案例做"案例不完整"报告，不对推断的数据做"数字证据缺失"报告）
3. 每处推断/待补充内容生成一条专属 suggestion：
   - dimension = `AI推断内容`
   - severity = `major`
   - problem = 口语说明："这里是 AI 推断的内容，演讲前需要替换成你的真实素材，否则上台讲时可能说不清楚"
   - options: A）已有真实素材，请直接替换（推荐）/ B）暂时保留，演讲前再确认 / C）删除此论点
4. 推断内容所在 Slide 的**非推断部分**仍正常审查（C 选 C 选 C 型，只跳过推断块本身）

### 六维度审查逻辑

**维度 1：空姐效应**

触发条件（满足任意一条）：
- PPT 正文段落 > 30 字且含完整句子结构（有谓语动词）
- PPT 正文与演讲口头说重叠度 > 50%（词集 Jaccard 相似度）
- 方法论型论点的 PPT 上只有名词堆叠，无任何结构

severity 判定：
- `critical`：PPT 正文 > 50 字且是连贯段落
- `major`：PPT 正文有 2+ 完整陈述句
- `minor`：个别句式偏书面但不影响大局

**维度 2：论点深度（Why层）**

触发条件（同时满足两条才触发）：
1. 演讲口头说 > 100 字（内容够充分，值得有 Why 层）
2. 找不到 Why 层信号词（因为/原因是/相比/本质上/之所以/而不是选）

severity 判定：
- `critical`：章节核心论点页（非背景/过渡页）无 Why 层
- `major`：普通论点页无 Why 层
- `minor`：Why 层存在但仅是"因为这样更好"类循环论证

**维度 3：真实案例完整性**

触发条件：
- 出现"某大厂/某公司/行业标杆"等匿名引用
- 案例结果只有定性描述，无数字
- 案例缺少冲突段（直接从背景跳到结果）

severity 判定：
- `critical`：核心论点的唯一案例完全匿名且无结果数字
- `major`：案例有名字但结果缺失，或有结果但缺冲突
- `minor`：案例完整但用知名大公司替代自身实践（距离感远）

**维度 4：假问题识别**

触发条件：
- 问题描述含"没有/不足/缺乏/不完善/无法"等否定词 + 名词短语的组合
- 特别关注开场问题引入和每章的痛点陈述

severity 判定：
- `critical`：开场问题或全场核心痛点是假问题
- `major`：章节问题引入是假问题
- `minor`：局部细节处有假问题表达

**维度 5：授人以渔**

触发条件：
- 核心分享论点只有 What/结果，没有"你可以这样做"
- 方法论的第一步是"建立文化/转变思维"等极度抽象描述

severity 判定：
- `critical`：整个演讲核心部分全是成果展示，无可复用方法论
- `major`：某核心论点缺乏可带走的方法论
- `minor`：方法论存在但第一步需要细化

**维度 6：数字/证据充分性**

触发条件：
- 重要论点使用"显著/大幅/明显/很大/大量"等模糊副词且无数字
- 成果陈述只有相对值（"提升 50%"）而无绝对值基线
- 痛点是主观感受而无客观数据支撑

severity 判定：
- `critical`：核心论点（开场痛点或结尾成果）无任何数字
- `major`：重要中间论点无数字，只有定性描述
- `minor`：有数字但只有相对值，缺绝对值基线

### Problem 字段写法规范（次需 I3）

**禁止写法（方法论术语）：**
- `"空姐效应违规：PPT 正文包含大段陈述性文字"`
- `"Why 层覆盖率不足，缺少因果推导链"`
- `"论点缺乏授人以渔维度"`

**正确写法（口语化真实后果）：**
- `"这页 PPT 上的字太多了，放映时听众会去读屏幕而不听你讲"`
- `"你说了'要建立 SLO 体系'，但没解释为什么——听众会在心里问：为什么是 SLO 而不是别的监控方式？"`
- `"你讲了你们做到了 99% 可用性，但听众离场后不知道自己能做什么"`

### Schema 校验

JSON 写入前用 schema 校验（`review-suggestion.schema.json`）；校验失败自动修正后重试，最多 3 次；只有校验通过才写入文件。

## 失败模式

### F1：误报健康内容（最危险）

**特征**：给每一页 Slide 都生成 suggestion，包括已经写得很好的页面；或对明显的视觉锚点（如 `before_after 布局：3周→4天`）也报空姐效应问题。

**危害**：讲者看到满屏建议后丧失信任，开始全部忽略，包括真正有价值的问题。误报比漏报更有害——宁可只报 3 条真实问题，也不要报 15 条掺杂噪音的建议。

**触发条件检测**：对一份高质量 slide-content.md（所有维度均合格）的测试用例，检查 suggestion 总数是否为 0 或极少（≤2 条）。

### F2：Options 缺乏具体内容

**特征**：options 字段写的是"建议精简 PPT 内容"或"可以增加具体数字"；problem 字段写"论点深度不足，缺乏 Why 层"。

**危害**：讲者看完建议不知道改成什么，需要自己重新想，建议的价值大幅打折。衡量标准：看完这条建议，讲者能立刻知道改成什么样吗？如果不能，建议无效。

**触发条件检测**：逐条检查 options 的 content 字段，是否含有"建议/可以/应该/需要/尝试"等方向词而无实际修改文字。

### F3：维度分类错误

**特征**：把 PPT 正文完整句子的问题的 dimension 填成 `"论点深度"`；或把案例匿名的问题填成 `"数字证据"`。

**危害**：review-dashboard.html 按维度筛选展示，分类错误导致讲者在筛选某维度时找不到对应问题。

**触发条件检测**：对照每条 suggestion 的 original + problem，检查 dimension 是否对应该问题的根本原因（不是表面现象）。

### F4：对 AI 推断内容做内容质量审查

**特征**：slide-content.md 中有 `[AI推断，请确认]` 标注的案例，AI 对其生成了"案例缺乏冲突段"或"真实案例不完整"的建议。

**危害**：推断内容本身就可能不准确，对它做内容深度审查会产生完全无意义的建议（建议改写一个根本不存在的案例）。

**触发条件检测**：在含有 `[AI推断]` 标注的测试用例中，检查是否有基于推断内容生成的六维度 suggestion（应该只有 `dimension="AI推断内容"` 的提示性建议）。

### F5：Severity 滥用 Critical

**特征**：大量问题都被标为 critical（超过总数 50%）；或 minor 问题（如个别句式偏书面）被标为 critical。

**危害**：critical 失去警示意义——当所有问题都"很严重"时，讲者不知道该先改哪个，实际上等同于没有优先级。

**触发条件检测**：检查 critical 占总建议数的比例，> 40% 为异常；检查已确认为 minor 的问题（如"数字只有相对值"）是否被标为 critical。

### F6：Source_lines 为估计值

**特征**：source_lines 中的行号是 AI 估计的，与文件实际内容不对应（如 slide-content.md 只有 80 行，但 source_lines.end = 156）。

**危害**：ppt-review apply 按 source_lines 定位修改内容，行号错误会导致 apply 修改到错误位置，破坏 slide-content.md 格式。

**触发条件检测**：验证每条 suggestion 的 source_lines.start/end 是否在文件实际行数范围内，且对应行的内容与 original 字段一致。

## 评分维度

### D1：信号精准度（25 分）

关联主需 **I1 信号精准**。这是 review-content 的核心价值所在。

**量化检测**：
- 给测试用例中已确认的真实问题全部报出 → 每漏一个 -4 分
- 给测试用例中已确认的健康内容生成 suggestion → 每误报一个 -5 分
- AI 推断内容未被六维度审查（只有专属提示 suggestion）→ +5 分保底

| 分数段 | 描述 |
|--------|------|
| 22-25 | 真实问题全部被发现；健康内容无误报；AI 推断内容正确跳过内容审查并生成专属提示；总数在合理范围（5-15 条） |
| 15-21 | 发现了主要问题，有 1-2 处误报或 1 处漏报；或 AI 推断内容被混入六维度审查 |
| 8-14 | 有 3+ 处误报，或漏掉了 2+ 个 critical/major 级问题 |
| 0-7 | 大量误报（>30% 针对健康内容）；或 suggestion < 3 条但文件有明显问题 |

### D2：方案具体性（25 分）

关联主需 **I2 方案具体**。每条建议的核心价值在于"改成什么"。

**量化检测**：
- 每条 suggestion 的 options.content 字段包含具体修改文字（非方向说明）→ 各 +5 分
- problem 字段无方法论术语（"空姐效应/Why层覆盖率"等）→ 整体 +5 分加成

| 分数段 | 描述 |
|--------|------|
| 22-25 | 所有 options 的 content 字段写出可直接采用的具体文字；problem 字段全部用口语描述真实后果；recommended 选项的 reason 具体（不是"推荐"两字） |
| 15-21 | 大部分 options 有具体内容，但有 1-3 条只写方向没有写出具体文字；problem 字段有 1-2 处用了方法论术语 |
| 8-14 | 超过 30% 的 options content 是方向说明；或 problem 字段普遍含方法论术语 |
| 0-7 | options 普遍缺乏具体内容；content 字段充斥"建议修改/可以考虑/建议增加"类表达 |

### D3：Severity 分级准确性（20 分）

关联次需 **I4 分级诚实**。

**量化检测**：critical 占总数 > 40% 为异常；任何已确认为 minor 的问题被标为 critical = -8 分

| 分数段 | 描述 |
|--------|------|
| 17-20 | critical 只给真正影响演讲可信度的核心问题（开场/核心论点）；major/minor 分级准确；critical 占比 ≤ 30% |
| 12-16 | 分级基本准确，有 1-2 条偏高（minor 标成 major，或 major 标成 critical） |
| 6-11 | critical 滥用（占比 >40%）；或所有问题都是 minor（无区分能力） |
| 0-5 | 分级完全混乱；或无 severity 字段 |

### D4：维度分类准确性（15 分）

关联次需 **I3 理由可信** + I5。

| 分数段 | 描述 |
|--------|------|
| 13-15 | 所有 suggestion 的 dimension 字段准确对应问题根本原因（按根本原因分，不按表面现象）；AI 推断内容统一归入 `AI推断内容` 维度 |
| 9-12 | 大部分分类准确，有 1-2 处跨维度混淆（不影响主要问题的可发现性） |
| 0-8 | 多处跨维度混淆；或所有建议集中在 1-2 个维度（其余维度全为 0 但文件质量一般）|

### D5：格式合规性（15 分）

关联次需 **I5 机器可读**。

**量化检测**（5 项，各 3 分）：

| # | 检测项 | 通过标准 |
|---|--------|---------|
