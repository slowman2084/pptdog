## 状态读取与结构解析

> **AI 必须完整执行以下5步，并打印结构摘要，再进入任何后续步骤。**

### 1. 确认项目 slug

```bash
# 确认项目 slug
_PROJECTS_DIR="$HOME/.pptdog/projects"
_SLUG_COUNT=$(ls "$_PROJECTS_DIR" 2>/dev/null | wc -l | tr -d ' ')

if [ "$_SLUG_COUNT" -eq 0 ]; then
  echo "⛔ 暂无项目，请先运行 /ppt-hours 新建项目"
  exit 1
elif [ "$_SLUG_COUNT" -eq 1 ]; then
  SLUG=$(ls "$_PROJECTS_DIR")
  echo "📁 自动选择唯一项目：$SLUG"
else
  echo "📂 检测到多个项目，请选择："
  ls -t "$_PROJECTS_DIR" | nl -ba   # 按修改时间倒序，最近优先
  echo ""
  echo "（直接回车选最近的项目，或输入编号）"
fi
```

### 2. 读取并解析 mindmap.md

```bash
cat ~/.pptdog/projects/$SLUG/mindmap.md 2>/dev/null \
  || echo "⚠️  未找到 mindmap.md，请先运行 /plan-mindmap"
```

AI 从 mindmap.md 中提取并内部记录以下结构（**不打印给用户，只作为内部索引**）：

```
章节列表：
  第 1 章：<章节标题>
    论点 1.1：<论点标题>
    论点 1.2：<论点标题>
    ...
  第 2 章：<章节标题>
    论点 2.1：<论点标题>
    ...
  （以此类推）
```

如果 mindmap.md 不存在，**立即停止**：
> ⛔ 请先运行 `/plan-mindmap`，再回来规划 Slide 内容。

### 3. 读取并解析 details.md（逐论点提取）

```bash
cat ~/.pptdog/projects/$SLUG/details.md 2>/dev/null \
  || echo "⚠️  未找到 details.md，建议先运行 /plan-details（内容会更丰富）"
```

AI 从 details.md 中，**逐论点**提取以下内容并内部记录（**这是后续所有生成的原料库**）：

```
论点 X.Y：<论点标题>
  证据/数据：<从 details.md §论点X.Y 的"证据/数据"字段提取>
  真实案例：
    背景：<...>
    冲突：<...>
    解决：<...>
    结果：<...>
  展开方式：<金字塔 / 一波三折 / F.A.B>
  深度笔记：<Why、推导逻辑、授人以渔要点>
```

> ⚠️ **关键规则**：后续每一个 Slide 的"演讲口头说"，必须直接引用此处提取的实际内容，**严禁使用任何占位符**（如「[具体内容]」「[演讲者根据实际情况填写]」等）。

### 4. 读取 ppt-hours.md

```bash
cat ~/.pptdog/projects/$SLUG/ppt-hours.md 2>/dev/null \
  || echo "(ppt-hours.md 不存在，将采用默认设定：分享型，30分钟)"
```

### 5. 扫描 references/ 目录

```bash
REFS_DIR="$HOME/.pptdog/projects/$SLUG/references"
if [ -d "$REFS_DIR" ]; then
  echo "=== references/ 目录内容 ==="
  ls -la "$REFS_DIR" 2>/dev/null | grep -E "\.(png|jpg|jpeg|webp|gif|svg)$" \
    || echo "(目录存在但暂无图片)"
else
  echo "references/ 目录不存在（可手动创建：mkdir -p $REFS_DIR）"
fi
```

### 5. 打印状态摘要

AI 打印以下状态摘要（**必须包含论点→内容覆盖率**）：

```
📋 项目：<slug>
📁 项目目录：$(readlink -f $HOME/.pptdog/projects/$SLUG 2>/dev/null || echo $HOME/.pptdog/projects/$SLUG)
📑 骨架章节数：<N> 章，<M> 个论点（来自 mindmap.md）
📝 详细内容数：<K> 个论点有内容（来自 details.md）
📊 内容覆盖率：<K>/<M>（若 K < M，需进入 Step 0 校验）
⏱  目标时长：<X> 分钟
🎯 演讲类型：<分享型 | 汇报型>
👥 听众：<听众描述>
🖼  references/ 可用图片：<N> 张
   [列出：文件名 | 关联论点（来自 details.md 中的引用）]
```

**依赖检查：**

| 文件 | 状态 | 影响 |
|------|------|------|
| mindmap.md | ✅ / ⚠️ 缺失 | 缺失则无法生成，必须先运行 /plan-mindmap |
| details.md | ✅ / ⚠️ 缺失 | 缺失则内容会较浅，建议先运行 /plan-details |
| ppt-hours.md  | ✅ / ℹ️ 不存在 | 不存在则采用默认设定，建议先运行 /ppt-hours |

## Step 0：mindmap ↔ details 对比校验

> **在正式生成前，必须先做对比校验，确保每个论点都有可用素材。**

AI 生成以下对比表：

```
论点 ID | 论点标题（来自 mindmap）  | details 是否有内容 | 图片引用           | 有哪些素材
--------|--------------------------|--------------------|--------------------|----------
1.1     | <论点标题>               | ✅ 有              | 📷 monitor-slo.png | 案例✅ 数据✅ 展开方式✅
1.2     | <论点标题>               | ⚠️ 有但较薄        | （无图）            | 仅有展开方式，无案例/数据
1.3     | <论点标题>               | ❌ 无              | （无图）            | —
2.1     | <论点标题>               | ✅ 有              | 📷 arch-deploy.png | 案例✅ 数据✅
...
```

**如有遗漏（❌ 无内容 或 ⚠️ 较薄）：**

**Qs-0：以下论点在 details.md 中内容不足，怎么处理？**

列出所有有问题的论点，然后给出选项：

```
有内容缺口的论点：
  - 论点 1.3「<论点标题>」：details 中无对应内容
  - 论点 1.2「<论点标题>」：仅有展开方式，无案例/数据

A. 现在补充——告诉我这些论点的实际内容（AI 结构化后继续）   ← 推荐，生成质量最好
B. 让 AI 推断——AI 根据上下文推断内容，标注「[AI推断，请确认]」，继续生成
C. 跳过这些论点——将这些论点从 Slide 规划中删除
D. 混合处理——部分补充、部分推断、部分跳过（告诉 AI 哪个怎么处理）
```

> 💡 选 A 生成质量最好，选 B 可以快速看全貌但需要事后确认 AI 推断内容。

**等待用户确认后，进入 Step 1。**

## Step 1：预计时长估算与超时预警

> 在生成任何 Slide 之前，先做时长估算。

**估算规则（AI 内置）：**
- 大开门 Slide：约 1.5 分钟
- 大关门 Slide：约 1 分钟
- 章节小开门 Slide（每章一个）：约 0.5 分钟
- 章节小关门 Slide（每章一个）：约 0.5 分钟
- 每个核心论点：
  - 轻量论点（仅数据，无案例）：约 1.5 分钟
  - 有案例论点（有背景/冲突/解决/结果）：约 3 分钟
  - 复杂论点（案例+数据+方法论推导）：约 4 分钟

**打印时长汇总表：**

```
模块             | 预计时长 | 备注
----------------|---------|----
大开门           | 1.5 min |
第1章 小开门      | 0.5 min |
论点 1.1 <标题>  | 3 min   | 有案例
论点 1.2 <标题>  | 2 min   | 纯数据
第1章 小关门      | 0.5 min |
第2章 小开门      | 0.5 min |
论点 2.1 <标题>  | 3 min   | 有案例
...              | ...     |
第N章 小关门      | 0.5 min |
大关门           | 1 min   |
----------------|---------|----
合计             | XX min  | 目标：<Y> 分钟
```

**超时提示：**
- 超目标 ≤10%：提示"略有超时，可轻微压缩"
- 超目标 10%-25%：🟡 警告，建议标注 [可压缩] 论点
- 超目标 >25%：🔴 严重超时，建议删减章节或合并论点

如有超时，**在继续前询问用户：**

**Qs-3：预计时长 {XX} 分钟，超出目标 {YY%}，怎么处理？**

```
A. 先生成完整版，生成后再决定删什么          ← 推荐，先看全貌
B. 现在就标出可压缩的论点，确认后再生成
C. 删掉最后一个章节（一般可省 20-30% 时间）
D. 自定义：我想删/合并某个具体章节（告诉 AI）
```

> 💡 超时 ≤15% 时选 A 通常没问题，现场可以语速稍快；超时 >25% 建议选 B 或 C，否则现场压力太大。

**等待用户确认后，进入 Step 2。**

## Step 2：大开门规划（全场整体）

> **开门目标：前 30 秒，让听众放下手机。**
> AI 必须基于 details.md 的实际内容，生成 2-3 个具体开门候选，**不是方向说明，是真实的文字**。

**AI 生成大开门候选方案（基于 details.md 的实际素材）：**

根据从 details.md 中找到的最有冲击力的素材（真实案例、反直觉数字、典型痛点），生成以下格式的候选（**每个选项必须写出实际内容，不是类型说明**）：

---

**Qs-1：以下是基于你的实际内容生成的大开门方案，选哪个？**

```
A. 【好故事】：「<基于 details.md 某个真实案例的开场句，含时间/地点/情绪细节>」
   → 用「<论点 X.Y 中的案例名/事件>」开场，前30秒不提解法，只讲当时的紧张和困境
   → PPT 上放：<一句话悬念，≤20字>
   → 预计时长：1.5 分钟

B. 【反常识】：「<基于 details.md 某个数字或结论，制造认知冲突的一句话>」
   → 用「<论点 X.Y 中的某个数据/对比结论>」，通过换算/对比方式制造"哇"的感觉
   → PPT 上放：<一个数字或短语，≤15字>
   → 预计时长：1.5 分钟

C. 【问题引导】：「<基于听众痛点和 details.md 内容，提出一个听众一定遇到过的困境>」
   → 用反问激发思考，聚焦在「<mindmap 核心主题>」的核心痛点上
   → PPT 上放：<一个反问句，≤20字>
   → 预计时长：1.5 分钟

D. 自定义：我有自己的开场想法（告诉 AI）
```

> 💡 推荐 <A/B/C（AI 根据素材强度判断）>，因为 <具体理由：例如"你的 details 里有 XX 数字，换算后效果最强"/ "第X章那个事故案例细节丰富，冲击力最大">。

**等待用户选择后，AI 生成完整大开门 Slide：**

```markdown
**PPT 上放：**
<一句话 / 一个数字 / 一个问题——引发好奇，不超过20字，不放答案不放结论，放"悬念">

<完整的开场内容，口语化，基于用户选定方案的实际素材>
```

## Step 3：主体 Slide 逐章节规划

> **对每一个章节，依次执行 3-a、3-b、3-c 三步，完成后进入下一章节。**
> 每个章节的小开门/小关门候选，**必须基于该章节 details.md 的实际内容**，不能是占位符。

---

### 对每个章节（循环执行）：

---

#### 3-a. 章节小开门

> **目标：让听众对本章产生好奇，不直接给答案，吊住注意力。**

AI 基于本章**第一个论点**（及后续论点）的素材，生成 2 个具体的候选小开门（**文字写出来，不是方向**）：

**Qs-章节N-开门：「第N章：<章节标题>」的小开门选哪个？**

```
A. 【悬念式】：「<基于本章第一个论点的实际数字/事实，制造悬念的一句话>」
   → PPT 上放：<≤15字的悬念句/数字>
   → 口头说：<30秒内的钩子，结尾引向"我们来看……">
   → 素材来源：（来源：details.md §论点N.1）

B. 【痛点式】：「<基于本章核心痛点，提一个听众一定有共鸣的困境>」
   → PPT 上放：<≤15字的问题/对比>
   → 口头说：<30秒内激活听众痛点，结尾引向"我们来看……">
   → 素材来源：（来源：details.md §论点N.1 或 N.2）

C. 自定义：我有自己的想法（告诉 AI）
```

> 💡 推荐 <A/B（AI 根据素材判断）>，因为 <具体理由>。

**等待用户选择后，AI 生成章节小开门 Slide：**

```markdown
## Slide — 第N章小开门：[本章钩子标题，≤15字]

**PPT 上放：**
<一个问题 / 一个数字 / 一个悬念——≤15字，不给答案>

**演讲口头说：**
<30秒内的钩子，口语化，基于选定候选的实际素材>
<结尾一句引向："我们来看……" / "这个问题，我们今天就来解决。">
预计时长：约 0.5 分钟

**配合说明：**
PPT 是悬念/问题，演讲者用声音钩住听众，不直接给本章答案。

**素材来源：** （来源：details.md §论点N.X）
```

---

#### 3-b. 本章各论点逐页 Slide

> **遵循以下设计原则（AI 在生成时严格执行）：**

**设计原则（内置，每页 Slide 生成时自动检查）：**

1. **空姐效应**：PPT 上不写你要读的内容。PPT 是视觉锚点，演讲是声音内容，两者互补不重复。
2. **架构图必须有红框/标注**：每张架构图都要注明"红框标注 XXX，说明 YYY"，强迫讲者明确"我要说这个图的哪个部分"。
3. **逻辑连接词口语化**：过渡句使用口语（"更进一步看" / "你有没有遇到过这种情况" / "这背后的原因是"），而不是书面语（"综上所述" / "接下来"）。
4. **每页最多3个信息点**：超过3个就拆成多页，或归并为上一层的标题。
5. **标题=论点**：检查每个标题——是话题（"监控建设"）还是观点（"监控覆盖率从30%到99%，故障定位时长降低90%"）？

**每个论点 Slide 的生成格式（演讲口头说必须引用对应论点的实际内容，绝不允许占位符）：**

> **PPT 内容设计原则：PPT 不是演讲内容的摘要，而是支撑论点的"视觉证据"。**
> 每页 PPT 的内容，应该让听众"一眼就信"——数字让人震，对比让人懂，图让人看见。
> AI 必须根据论点类型，选用对应的布局推荐（见下方），在「PPT 上放」字段里
> 写出"布局名 + 具体内容"，不能只堆关键词。
>
> 完整布局词汇表参见：`skills/slide-content-and-scripts/references/ppt-layout-vocabulary.md`

**论点类型 → 推荐布局（AI 必须按此选用，写出"布局名 + 具体内容"）：**

| 论点类型 | 推荐布局 | 备选布局 | 写法要点 |
|---------|---------|---------|---------|
| **数据型**（用数字证明成果/效果） | `before_after` / `big_number` / `two_stat`/`three_stat` | `metric_cards` / `waterfall` | 数字要有对比（前→后）+ 数据来源；只突出最震撼的那个 |
| **案例型**（用真实故事证明观点） | `timeline` / `case_study` / `before_after` | `content_right_image` | 放冲突点+解法，不放完整故事；高亮转折词 |
| **方法论型**（讲一套可复用做法） | `process_chevron` / `vertical_steps` / `icon_grid` | `temple` / `cycle` / `checklist` | 步骤用动词短语（≤6字），有先后顺序用箭头 |
| **原理/因果型**（解释为什么） | `venn` / `decision_tree` / `two_column_text` | `pyramid` / `pros_cons` | 把推导路径可视化；高亮关键变量/决策点 |
| **架构/系统型**（展示系统全貌） | `content_right_image`（含架构图）/ `dashboard_kpi_chart` | `icon_grid` | 必须有红框标注聚焦模块；只说一个核心模块 |
| **对比型**（我们 vs 别的做法） | `side_by_side` / `metric_comparison` / `pros_cons` | `swot` / `harvey_ball_table` | 维度要中立；胜出项高亮；必须有对比列 |
| **状态汇报型**（项目/KPI 健康度） | `scorecard` / `rag_status` / `dashboard_kpi_chart` | `metric_cards` / `kpi_tracker` | 红/橙/绿状态一目了然；配图表展示趋势 |

```markdown
## Slide N — [标题（=论点，≤21字，含数字/痛点）]

**论点类型：** <数据型 / 案例型 / 方法论型 / 原理型 / 架构型 / 对比型 / 状态汇报型>

**PPT 上放：**
布局：<布局名>（<布局中文名>）
（按上方推荐布局，写出具体内容——不是描述说明，是真实数据/步骤/对比项）

示例（数据型 → before_after）：
```

### 5-B：标题论点检查

遍历所有 Slide 标题，检查：
- 是话题（❌ "监控建设"）还是论点（✅ "监控覆盖率提升至99%，故障定位提速90%"）？
- 有没有废话标题（❌ "发现了一个问题" / ❌ "通过XX策略，突出优势"）？

打印需要修改的标题列表，询问用户确认后更新。

### 5-C：空姐效应最终扫描

检查每页 PPT 的"PPT 上放"部分：
- 有没有完整句子（应该是关键词/数字/图示）？
- 有没有和演讲口头说的内容重叠超过50%？

打印有问题的 Slide 编号，建议简化方式。

### 5-D：逻辑连接词检查

检查每页的"承接上一页"过渡句：
- 有没有 Slide 之间逻辑跳跃（前后无连接）？
- 过渡句是否口语化？

### 5-E：主语检查（用"我"不用"我们"）

> 汇报型专项检查（分享型也适用，但汇报型必须执行）

对所有 Slide 的**演讲口头说**部分，逐一扫描是否出现"我们"作为主语：

| 出现位置 | 原句 | 建议改写 |
|---------|------|---------|
| Slide N | 我们发现了… | 我发现了… |
| Slide M | 我们的方案是… | 我的判断是… / 我决定… |

**常见替换规则：**
```
❌ 我们发现 / 我们认为 / 我们做了 / 我们的成果
✅ 我发现 / 我判断 / 我做了 / 我主导的成果

❌ 我们团队 → ✅ 我（负责人）/ 我带领的团队（若需要说明规模）
❌ 我们决定 → ✅ 我的决策是 / 我选择了
```

若整个 slide-content.md 中没有出现"我们"作为主语 → 打印：`✅ 主语检查通过`
若有出现 → 列出位置和建议改写，询问用户：
```
Qs-主语：以上"我们"建议改为"我"，如何处理？
A. 全部自动改为"我"（AI 帮改）        ← 推荐
B. 逐条确认（AI 逐一问）
C. 保留"我们"（我知道风险，不改）
```

### 空姐效应（核心原则）
> 眼睛盯着 PPT 就离不开了。PPT 上写了的，演讲时就不要再读。

- PPT 是视觉锚点：关键词、数字、对比图
- 演讲是声音内容：Why、故事、情感共鸣、方法论

### 标题=论点（电梯法则）
> 听众在电梯里看到这个标题，3秒内应该能明白你在说什么重要的事。

```
❌ 话题型标题：
  "监控建设"
  "质量体系优化"
  "K8S 不满足我们的诉求"

✅ 论点型标题：
  "监控覆盖率从30%到99%，故障定位时长降低90%"
  "三个措施将P99延迟从2s降至200ms"
  "容器化后资源复用率提升60%，省下XX台服务器"
```

### 开门/关门对应关系
| 演讲特征 | 推荐大开门 | 推荐大关门 |
|---------|----------|----------|
| 有真实故事、情绪弧度 | 好故事 | 升华（首尾呼应） |
| 听众有明确痛点 | 问题引导 | 流程图（给解决方案） |
| 有反直觉的数据 | 反常识 | 重复（加深印象） |
| 系统性框架/全景 | 问题引导 | 地图 |
| 汇报型、结果导向 | 反常识/数字冲击 | 重复 |

### 每页Slide信息量控制
```
1个信息点 = 1个关键词 / 1个数字 / 1张图
每页最多3个信息点

若内容超过3点：
→ 检查是否可以拆成2页
→ 检查是否可以归并为1个上层概念
→ 不要"硬塞"：多即是少
```

### 逻辑连接词口语化示例
```
❌ 书面语：综上所述 / 由此可见 / 接下来 / 此外
✅ 口语化：
  "更进一步看，我们发现……"
  "你有没有遇到过这种情况……"
  "这背后的真正原因是……"
  "但这只是第一层，更深的是……"
  "所以问题来了……"
  "带着这个结论，我们再看……"
```
